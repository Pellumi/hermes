Autonomous Drone Operating System (ADOS)
Interface Control Document (ICD)


Version: 1.0
Phase: Simulation + Supervised Autonomy
Deployment Geography: Abuja Pilot
Status: Final - Approved for Implementation


1. Purpose
This document defines the interfaces, communication protocols, data contracts, and interaction patterns between all components of the Autonomous Drone Operating System.


The ICD ensures that:
* Edge systems (drones) integrate correctly with ADOS
* External logistics platforms interact with ADOS safely
* Internal ADOS services communicate reliably
* Telemetry and command flows remain deterministic
* System modules remain independently replaceable
* Local container-drone communication is secure and fast
* Recipient communication (calls/SMS/WhatsApp) is standardized
* All 50+ events defined in Data Architecture are emitted


2. System Interface Topology
ADOS operates as a distributed system consisting of:
Layer
	Components

5.3 Local Command Payload (Container → Drone)
Characteristic: command_rx
{
 "command_type": "WEIGHT_VERIFIED_LOCAL",
 "timestamp": "2026-03-17T10:32:15Z",
 "sequence_number": 42,
 "weight_kg": 2.3,
 "threshold_kg": 8.0,
 "sensor_status": "OK",
 "hmac_signature": "sha256_hash_of_payload"
}
	

Command Types:
* WEIGHT_VERIFIED_LOCAL - Weight <= 8kg
* WEIGHT_REJECTED_LOCAL - Weight > 8kg
* CONTAINER_FAULT - Sensor or mechanism error
* HEARTBEAT - Periodic status (10s)


5.4 Drone Acknowledgement (Drone → Container)
Characteristic: command_tx
{
 "ack_status": "ACCEPTED",
 "command_sequence": 42,
 "timestamp": "2026-03-17T10:32:15Z",
 "processing_time_ms": 12,
 "next_action": "LOCK_CONTAINER"
}
	Ack Status Values: ACCEPTED, REJECTED, RETRY_LATER, FAULT


5.5 Weight Stream
Characteristic: weight_stream
Frequency: 1Hz during loading operations
{
 "weight_kg": 2.3,
 "timestamp": "2026-03-17T10:32:15Z",
 "sequence": 42,
 "stability": "STABLE"
}
	Stability Values: STABLE, FLUCTUATION, ERROR


5.6 Security
* Each container-drone pair shares a unique 256-bit key provisioned during first pairing 
* All commands include HMAC-SHA256 of payload
* Sequence numbers prevent replay attacks
* Keys rotated every 30 days via secure OTA update

9. Telemetry Ingestion Interface
Telemetry pipeline receives drone telemetry and validates before storage


9.1 Pipeline Flow
Drone → MQTT Broker (TLS) → Validation Service → TimescaleDB
                                              → Redis Cache
                                              → WebSocket Broadcast
                                              → Kafka Events
                                              → Dead Letter Queue (on failure)
	

9.2 Validation Rules
Field
	Validation Rule
	Action on Failure
	timestamp
	Within ±5s of server time
	Reject to DLQ
	sequence_number
	> last sequence for drone
	Reject to DLQ
	hmac _signature
	Matches computed HMAC
	Reject to DLQ, flag security incident
	battery_percent
	0-100 range
	Clamp, log warning
	location
	Within Abuja bounding box or simulation flag
	Accept, flag for review
	state
	Valid mission state
	Reject, operator alert
	

9.3 Dead Letter Queue Schema
Topic: telemetry/dead_letter
{
 "original_message": { ... },
 "failure_reason": "INVALID_HMAC",
 "failure_timestamp": "2026-03-17T10:30:01Z",
 "drone_id": "DRN_001",
 "retry_count": 0,
 "next_retry": "2026-03-17T10:31:00Z"
}
	Retry Policy: Exponential backoff: 1min, 5min, 30min, 2h, 6h, then permanent failure

15.2 Drone Certificate Rotation
* Certificates valid for 90 days
* Renewal request 30 days before expiry
* New certificate provisioned via secure OTA update
* Old certificate accepted for 48 hours after rotation


15.3 HMAC Key Rotation (Container - Drone)
* Keys rotated every 30 days
* New key provisioned during maintenance docking
* Old key accepted for 24 hours after rotation

18. Interface Versioning
18.1 Versioning Strategy
All interfaces follow semantic versioning: MAJOR.MINOR.PATCH
* MAJOR: Breaking changes (new endpoint versions)
* MINOR: Backward-compatible additions
* PATCH: Backward-compatible fixes


18.2 API Versioning
REST APIs versioned in URL path:
/api/v1/missions
/api/v2/missions
	

18.3 MQTT Topic Versioning
Topics include major version:
drone/v1/{drone_id}/telemetry
warehouse/v2/{warehouse_id}/dock_requests
	

18.4 Event Versioning
Event envelope includes version field:
{
 "event_type": "mission.started",
 "version": "1.0",
 ...
}
	

18.5 Deprecation Policy
* Deprecated endpoints continue to work for 6 months
* Deprecation announced via:
   * Response header: X-API-Deprecated: true
   * Developer documentation
   * Email notification to registered contacts
